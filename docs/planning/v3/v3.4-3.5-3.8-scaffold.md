# v3.4 / v3.5 / v3.8 — accounts, Spotify, notifications (scaffold)

> Phase plan: [`PLAN.md`](./PLAN.md). These three segments need **live external
> services + product wiring** that can't be exercised offline, so v3 lands their
> **data layer + tested cores + seams** here and defers the live wiring to operator
> steps. Nothing below is a stub that pretends to work — each piece is real and
> compiling/tested; the gaps are explicitly the parts that require secrets + a
> running browser session.

## What landed (real, gated)
- **DB** — `db/migrations/20260626_v3_4_user_data.sql`: `user_favorites`,
  `user_playlists`, `user_playlist_items`, `notification_prefs`. Owner-scoped RLS
  (`auth.uid() = user_id`); additive + reversible.
- **Auth seam** — `lib/auth.ts`: browser auth client + `signInWithSpotify` /
  `signInWithEmail` / `signOut` / `getCurrentUser`. Empty-safe (no keys → no-ops).
- **Change detection** — `pipeline/notify.py`: `lineup_delta` + `notify_events`
  (pure, self-tested). The notification core minus delivery.

## Operator steps to go live (deferred)

### v3.4 — accounts
1. Enable **Supabase Auth** in the project; apply the migration.
2. Add `@supabase/ssr` and a server client for cookie-based session reads (the one
   piece `lib/auth.ts` intentionally omits — server RLS needs the user's JWT).
3. UI: sign-in button (Nav), account menu, and `mergeLocalFavoritesOnSignIn` —
   flush device-local IndexedDB favourites (`lib/favorites.ts`) into `user_favorites`
   on first sign-in (merge, don't clobber). Keep signed-out working (local fallback).

### v3.5 — Spotify deepening
1. Supabase Auth → enable the **Spotify provider**; set redirect
   `${origin}/auth/callback`. `signInWithSpotify` already requests the playlist +
   top-read scopes.
2. Verify user-token track resolution (the v2.11 follow-up) end-to-end; persist
   saved-playlist history; handle token refresh/revoke (Supabase manages provider
   token refresh when the provider is configured).

### v3.8 — notifications
1. Wire `pipeline/notify.py`: snapshot each festival's lineup per `ingestion_runs`,
   `lineup_delta` vs the previous successful run, `notify_events` against
   `user_favorites`.
2. Delivery: **web push** (VAPID keys → `notification_prefs.web_push_subscription`)
   and/or **email** (provider key). Respect per-user `channel` + opt-in flags.
3. Add a daily `detect-lineup-changes` cron job once delivery exists.

## Why deferred, not faked
The project's gate is "only a closed phase merges to main." Shipping a fake
sign-in button or a no-op push sender would be worse than honest scaffolding: it
would look done without being verifiable. The tested cores (auth seam, change
detection) + the schema are the parts that *can* be proven offline; they're done.
