# v3.10 / v3.11 / v3.12 — performance, observability, deploy gates

> Phase plan: [`PLAN.md`](./PLAN.md). Three infra segments grouped: they share the
> "make it fast + observable + gated at scale" goal and touch config/dashboards
> rather than product surface.

## v3.10 — Performance at scale
- **ISR** on the hot detail pages: `festival/[slug]` and `artist/[slug]` switched
  from `force-dynamic` to `export const revalidate = 600`. At catalog scale this is
  a large TTFB win — pages are cached and refreshed every 10 min, matching the daily
  cron's update cadence (acceptable staleness).
- **Image pipeline**: `next.config` `images.minimumCacheTTL = 86400` + AVIF/WebP
  formats — optimized art is cached a day and served in modern formats.
- *Deferred:* cache-tag invalidation on ingest (needs a revalidate webhook from the
  pipeline) and the v2.10.1 edge-runtime revisit — both want real Web-Vitals data first.

## v3.11 — Observability & reliability
- **`/status` dashboard** reads the `ingestion_runs` run-log (public-read RLS):
  recent runs, per-festival freshness, stale (>30d) + error counts. The same data
  the v3.1 gate enforces, now visible. ISR 5 min.
- *Deferred:* external error tracking (Sentry-style) + alerting — needs a DSN/secret;
  the dashboard is the offline-buildable half.

## v3.12 — Deploy-gated verification (v2 carryover)
Turning v2's "tracked follow-ups" into real gates:
- **Lighthouse / Web Vitals** — `.lighthouserc.json` budgets (perf/a11y/CLS/SEO) +
  `.github/workflows/lighthouse.yml` (build → serve → `@lhci/cli autorun`) on PRs.
  Secret-free (offline render is enough for the shell budgets). **Wired.**
- **`/canary`** post-deploy, **PWA install/offline e2e**, **Spotify login→playlist
  e2e** — documented gates that need a deployed URL + test creds; they run against
  a real deployment, not in PR CI. Carry into the live deploy pipeline. **Documented.**

## Why grouped + partly deferred
These are infra: the offline-buildable parts (ISR, image config, the status
dashboard, the Lighthouse budget) are done and gated; the parts needing live
deploys/secrets (canary, e2e, error DSN) are documented operator steps, consistent
with the phase's "merge only what's green" rule.
