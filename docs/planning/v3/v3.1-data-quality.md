# v3.1 — Data quality & backfill (segment plan)

> Phase plan: [`PLAN.md`](./PLAN.md) · Workflow: [`docs/WORKFLOW.md`](../../WORKFLOW.md)
> Branch: `v3.1-data-quality` (off `v3`). Prior segment: [`v3.0-foundation.md`](./v3.0-foundation.md).

## Goal
Clear the data debt the v2.3 live validation surfaced, and make data quality an
**enforced gate**, not a tracked note — so the v3.2 many-festival push starts from
a clean, self-policing catalog.

## Why now (current state at kickoff)
The live validator reported **22 errors** across the seeded non-Lolla stub rows:
20 festivals missing dates, 2 missing coordinates, plus 2 accent-folding artist
duplicate groups from before the v2.3.2 slug fix. Lolla (the shipped festival) was
clean; the debt was all in v3-staging stubs.

## Decisions taken
- **Dates** — flag all 20 `dates_estimated` (incl. defunct ones: panorama, fyf-fest,
  pemberton, meadows, lakeshake). Real dates land during v3.2 ingestion; the flag is
  honest provenance for a stub. Defunct rows were *not* deleted (carry to v3.2 scope).
- **`punk-in-drublic`** — deleted. A touring fest with `venue=Various`/`city=Various`
  has no geocodable location and coords have no `estimated` escape hatch; the row was
  an empty stub (0 lineups/0 stages), so deletion lost nothing.
- **Dedupe keeper** — kept the *canonical-slug* row (not the row holding the cache, as
  the v2.3.1 hint suggested), then moved the dupe's lineup + cache onto it. Keeps the
  correct slug the frontend resolves on. No slot-key collision: in both groups the
  keeper and dupe lineups sat at different festivals.

## Freshness semantics (the non-obvious bit)
`stale = a festival we actively ingest has had no successful run in N days` (default 30).
A festival with **zero** `ingestion_runs` rows has *no freshness contract yet* — it's a
manual seed, skipped, not flagged stale. Without this scoping, the ~40 manually-seeded
festivals would false-red the gate the moment it turned on. Freshness activates per
festival as v3.2 brings it under adapter management.

## Freshness depends on v3.0's live apply
The freshness query reads `ingestion_runs`, created by
`db/migrations/20260625_v3_0_1_source_registry_and_run_log.sql` — a v3.0 artifact whose
**live apply is a pending operator step** (see v3.0-foundation.md → "Apply order"). Until
that migration runs on the live DB, `_check_freshness` degrades gracefully: it prints a
loud `⚠ freshness skipped: ingestion_runs missing` and does **not** red-fail the data
gate (a missing foundational table is an ops gap, not a data regression). Run the v3.0
apply order to activate freshness.

## CI wiring
- `ci.yml` (PR gate, secret-free): `validate_data.py --self-test` — unchanged, still the
  blocker for pull requests, now also covering the `stale_festivals` pure logic.
- `etl_daily.yml` (live gate, has secrets): new `validate-live` job runs
  `validate_data.py --max-age-days 30` after the write jobs (`enrich-artists`,
  `sync-spotify`, `enrich-locations`). A red here fails the daily ETL — the live
  counterpart to the offline self-test. Live validation can't run in PR CI (no secrets;
  fork PRs can't safely access them), so the gate splits by capability.

## Exit criteria
- ✅ Live validation green across all live festivals (76/76; was 22 errors).
- ✅ Dedupe report clean (0 duplicate groups, 0 mangled slugs).
- ✅ Validation is an enforced gate: offline self-test blocks PRs; live + freshness
  blocks the daily ETL.
- WORKFLOW §2 gates green; segment merged to `v3`, sub-branch deleted.

## Scope guard (ponytail)
v3.1 clears existing debt and arms the gate. It does **not** backfill *real* dates for
active fests, fix the `boots-hearts` `country=US`→`CA` scope error, or model touring
fests — those are v3.2 (ingestion at scale) concerns. The flag/delete/gate is the
minimum that makes the catalog valid and self-policing today.
