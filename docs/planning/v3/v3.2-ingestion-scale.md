# v3.2 — Multi-festival ingestion at scale (segment plan)

> Phase plan: [`PLAN.md`](./PLAN.md) · Workflow: [`docs/WORKFLOW.md`](../../WORKFLOW.md)
> Branch: `v3.2-ingestion-scale` (off `v3`). Prior segments:
> [`v3.0-foundation.md`](./v3.0-foundation.md), [`v3.1-data-quality.md`](./v3.1-data-quality.md).

## Goal
Take the catalog from 76 → 200+ US festivals, each passing the v3.1 validation gate
(real dates **or** `dates_estimated`, venue/city, coords) and staying fresh (<30d once
ingested) — **without** the daily cron timing out at full scale and **without** writing
one Python script per festival.

## Decision (resolved at kickoff)
The v3 open question "per-site scrapers vs paid aggregator API vs hybrid" is settled:

- **Hybrid.** A free multi-aggregator fills the metadata *spine* for all ~200 (name,
  dates, venue, coords — exactly the gate fields) through **one** adapter; curated
  `manual_config` + a few ported scrapers give lineup/schedule *depth* on a flagship tier.
- **Free tiers only.** Ticketmaster Discovery API (primary; already wired in
  `lineup_scraper.py`) **+** SeatGeek (optional, cross-validation). Designed so a third
  provider slots in behind the same `AggregatorAdapter`.
- **Curated target list**, not blind discovery — we only onboard rows we can satisfy,
  so the gate stays green. Seeded from the existing 76 + `lineup_scraper.FESTIVAL_CONFIG`.
- **Coalesce-forward, trust-ordered merge** — aggregator fills missing fields and
  replaces *estimated* dates, but never clobbers curated/official metadata.
- **Exit bar:** all ~200 pass the gate on metadata; a flagship tier (~10–20) additionally
  carries full lineup + schedule. Lineups are **not** required for the gate.

Why this shape: the v3.1 gate needs dates/venue/coords — an aggregator delivers exactly
those at scale, one adapter for many festivals. Lineup/schedule depth is the part that's
expensive and brittle to scale, and it only matters deeply for flagships, so it stays
curated. The v3.0 `SourceAdapter` contract, `ingestion_runs` run-log, per-source
`lineups.confidence`, and the v2.3.3 trust triggers already exist to merge these tiers.

## Architecture

### Target list — `pipeline/festival_targets.csv`
`slug,name,city,state,tier,tm_keyword,seatgeek_query` — the curated set we want. Data,
not code; editable in a PR. `load_targets()` parses it; the aggregator resolves each
target by keyword/venue against the providers. Seeded with the ~26 real flagships from
`lineup_scraper.FESTIVAL_CONFIG`; grows toward 200 as a data task.
- *ponytail:* CSV over a table — no migration, no RLS, pipeline-internal. Promote to a
  `festival_targets` table only if the frontend/UI ever needs it.

### `AggregatorAdapter` (new, subclasses `SourceAdapter`)
One adapter; providers behind a small strategy:
- `TicketmasterProvider.search(target)` and `SeatGeekProvider.search(target)` →
  candidate dicts `{name, venue, city, state, latitude, longitude, start_date, end_date}`.
  tenacity-wrapped HTTP, rate-limit-aware. SeatGeek degrades to no-op without its key.
- `fetch()` runs the enabled providers **for one target festival** (so each run logs one
  `ingestion_runs` row keyed to that `festival_slug` — required for per-festival freshness).
- `normalize()` **cross-merges** provider candidates: agreement → high confidence;
  one-sided → fill the gap; date conflict → prefer Ticketmaster (primary) and count the
  disagreement into `stats` (no per-festival confidence column — kept in the run-log).
- Writes **festival metadata only** — no lineups (provider lineup depth is too shallow).
- `festival_merge = "coalesce"` attribute tells the orchestrator to merge, not clobber.

### Coalesce-forward merge — `store.merge_festivals(rows, trust)`
The generic `run()` festival path clobbers (`set name=excluded.name…`). For non-official
sources the orchestrator branches to `merge_festivals`, which, per slug:
- **dates:** set real dates **and clear `dates_estimated`** iff existing is null OR
  `dates_estimated=true`; never overwrite curated/official dates.
- **latitude/longitude/venue/city/state/timezone:** fill only when existing is null/empty.
- **never touched:** `name`, `accent_color`, `tags`, `website_url`, `wikipedia_url`,
  `description`, social handles, `hero_image_url`, `logo_url`, `vector_art`.
- New target not yet in DB → insert. Provider returned nothing usable → skip (don't
  insert a gate-failing stub).

This backfills the 22 `dates_estimated` stubs + missing coords without stomping anything
hand-tuned, and needs **no new per-field-trust columns** (reuses `dates_estimated` +
null-ness as the provenance signal).
- *ponytail / alt:* explicit per-field provenance columns — heavier; deferred until a
  second authoritative source actually conflicts.

### Batch runner + staleness rotation — `pipeline/ingest_festivals.py`
- `--batch N` → process the **N stalest** targets (oldest successful `ingestion_run`,
  never-ingested first). `select_stale_targets()` is a pure, tested function.
- `--all` → full backfill (batched, off the daily cron). `--festival SLUG` → one.
- Loops chosen targets, calling `run()` once per target → one `ingestion_runs` row each.
- Rotation bounds API calls + cron time by construction: M≈30/day cycles all ~200 inside
  the 30-day freshness window, so we never refresh all 200 every day.

### Cron sharding — `.github/workflows/etl_daily.yml`
- **Aggregator refresh:** single job, `ingest_festivals.py --batch 30` (fast, rotation-bounded).
- **Flagship scrapers:** `strategy.matrix.shard:[0..3]`, each handles a slice — parallel,
  no timeout.
- v3.1 `validate-live` job still runs after the write jobs and gates the post-ingest state.

### Flagship depth proof
Port the Ticketmaster lineup fetch (`lineup_scraper.fetch_tm_lineup`) onto a
`SourceAdapter` (`TicketmasterLineupAdapter`) that writes **lineups** with per-source
confidence — proving the scrape/API *lineup* path on the v3.0 contract. The v2.3.3 trust
triggers referee it vs. curated rows. We do **not** build 200 scrapers.

## Testing
Assert-based offline self-tests (no DB, no network, no keys — like `test_ingest.py`):
provider `normalize` over fixture JSON · cross-source merge (agree / fill / conflict) ·
`merge_festivals` (estimated→real dates + flag cleared, official preserved, null coords
filled, non-null kept, new-row insert) · target-CSV parse · `select_stale_targets`
ordering. Live apply (needs API keys + service-role) is an operator step.

## Apply order (operator, live DB)
1. `.env`: add `TICKETMASTER_API_KEY` (already used by `lineup_scraper`), optional
   `SEATGEEK_CLIENT_ID`.
2. `db/seed_sources.sql` (now includes the `festival-aggregator` source row).
3. `cd pipeline && python ingest_festivals.py --all` — one-time backfill (batched).
4. Daily cron then keeps the catalog fresh via the rotation job.

## Exit criteria
- ≥ 200 festivals (or as many as the curated target list holds), each passing the v3.1
  gate; flagship tier carries full lineup + schedule.
- Daily cron completes within budget (rotation + matrix sharding).
- Aggregator runs log to `ingestion_runs`; freshness activates per festival as it comes
  under management.
- WORKFLOW §2 gates green; segment merged to `v3`, sub-branch deleted.

## Scope guard (ponytail)
v3.2 builds the aggregator + merge + rotation and proves lineup depth on one flagship. It
does **not**: curate all 200 target rows (data task; the machinery is list-length-agnostic),
add per-field provenance columns, write provider lineup ingestion, or model touring
festivals. Those are follow-ons, deferred to v3.2.7 / later segments.
