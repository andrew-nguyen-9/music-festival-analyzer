# v2.3.1 — Scraper Accuracy Audit (Lollapalooza 2026)

Ground truth: the hand-curated `pipeline/lolla_schedule_seeder.py` `SCHEDULE`
(8 stages × 4 days = ~210 sets, with stage, day, start/end times). It is the
*trusted* path. The generic scrapers are the *automated* path; this audit
compares them against the curated truth and enumerates error classes that the
rest of the v2.3 segment addresses.

## Sources in play

| Script | Source(s) | Produces | Trust |
|--------|-----------|----------|-------|
| `lolla_schedule_seeder.py` | hand-curated | stage + day + set times, per artist | **high** (official poster) |
| `lineup_scraper.py` | Ticketmaster, Setlist.fm | artist list, day (TM only), headliner-by-billing | medium / low |
| `schedule_scraper.py` | Ticketmaster, Songkick, Claude OCR | stage + day + set times | low–medium |
| `festival_dates_enricher.py` | Ticketmaster, Wikipedia, historical pattern | festival start/end + `dates_estimated` | medium |
| `festival_scraper.py` | hardcoded dict | city/state/venue (no lat/lng) | high (static) |

## Error classes found

1. **Slug mangling on accented names (data-integrity bug).**
   The seeder shipped its *own* `slugify()` that strips any non-`[a-z0-9]`
   char to `-` with **no transliteration**. `lineup_scraper`/`schedule_scraper`
   use `python-slugify`, which transliterates. Result: the same artist gets two
   different slugs depending on which script wrote it —
   `Röz` → `r-z` (seeder) vs `roz` (python-slugify); `Adéla` → `ad-la` vs `adela`.
   Two slugs ⇒ two `artists` rows ⇒ split lineups ⇒ a broken Spotify-cache join
   for one of them. **Root cause:** no single canonical slug function.
   → fixed in v2.3.2 (`pipeline/names.py`, all writers route through it).

2. **Duplicate / near-duplicate artist rows.**
   Beyond slug drift: billing-name variants (`BBNO$` vs `bbno$`, `I-DLE` vs
   `(G)I-DLE`), and genuine repeats across days are fine, but cross-script
   variants are not. → v2.3.2 ships a conservative **dedupe reporter** (flags,
   never auto-merges across FK tables in cron).

3. **Ticketmaster lineup noise.**
   `lineup_scraper` accepts a TM "event" as a festival day only if it has ≥5
   attractions and isn't an obvious aftershow — heuristic, lets through
   promoter side-events and misses days TM hasn't published. TM rarely carries
   per-set stage/time for festivals, so it yields a *name list*, not a schedule.

4. **Setlist.fm has no per-day / per-stage structure.**
   The fallback returns a flat artist list with `day=None`, `stage=None`,
   `set_time=None`. Usable for "who played" historically, useless for the
   schedule grid v2.8 needs.

5. **Missing set times & stages on the automated path.**
   Only the curated seeder and (sometimes) Songkick/OCR carry `set_time_start/end`
   and `stage`. Most rows the scrapers write have null stage/time. → v2.3.3
   makes the curated path authoritative for Lolla and the validator enforces
   completeness.

6. **Timezone ambiguity.**
   `lineups.set_time_*` is a bare `time` with no zone, and `festivals` had no
   timezone. A 23:00 set on `2026-08-01` is meaningless without knowing it's
   `America/Chicago`. Storing UTC would corrupt all-day local schedules.
   → v2.3.3 adds `festivals.timezone` (IANA) and keeps set times **local**.

7. **No coordinates anywhere.**
   `festivals` had city/state/venue but no lat/lng; there was no stage geometry
   at all. v2.8's phone-background needs both. → v2.3.4 adds geocoded
   festival lat/lng (Nominatim) + a `stages` table.

8. **Nominatim cannot resolve intra-park stages (provider limitation, not failure).**
   Nominatim resolves *venues/cities* ("Grant Park, Chicago") fine, but it has
   no notion of "T-Mobile Stage". So: geocode the **venue** automatically;
   seed **per-stage** coordinates from the festival's published layout. This is
   expected, not a reason to swap providers — Nominatim resolves the real
   venues we tested. (Provider stays Nominatim per the locked decision.)

9. **Dates: estimated vs confirmed conflation.**
   `festival_dates_enricher` already flags `dates_estimated`, but nothing
   *validated* that an estimated range is internally sane or that lineup `day`s
   fall inside the festival's start/end. → v2.3.5 validator closes this.

## Reconciliation / conflict policy (locks an open question)

Trust order, highest wins:

```
official/curated  >  wikipedia  >  ticketmaster / songkick  >  estimated
```

- Provenance is recorded per lineup row in the new `lineups.source` column.
- A row is **verified** iff `source ∈ {official, wikipedia}`; the UI flags
  anything else as unverified.
- Higher-trust sources overwrite lower-trust on conflict; a lower-trust source
  never overwrites a verified row (the writers check before downgrading).
- Wikipedia is used to *fill gaps and cross-check*, not to override the official
  poster.

## Stage-coordinate schema (locks the other open question)

A dedicated `stages` table, `unique (festival_id, name)`, holding
`latitude/longitude` + a `coords_source` flag. v2.8 consumes it by joining
`lineups.stage = stages.name`. Chosen over a per-`lineups`-row coordinate
(which would duplicate the same point across every set on that stage).
